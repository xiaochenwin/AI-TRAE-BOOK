
### 9.1 Plan模式的设计理念

在软件开发的历史长河中，先规划后执行一直被视为最佳实践。从瀑布模型到敏捷开发，无论方法论如何演进，规划这一环节始终不可或缺。然而，在AI编程工具兴起之后，一种新的工作模式开始流行：直接告诉AI你想要什么，然后让它直接生成代码。这种模式虽然快捷，但在面对复杂需求时往往暴露出种种问题。

SOLO的Plan模式，正是对这一问题的深刻回应。它的设计理念可以概括为三个关键词：可控性、透明性和纠错性。


#### 9.1.1 可控性：让AI不再是一个黑盒

直接执行模式下，AI接收到需求后就开始生成代码，用户在这个过程中几乎没有任何控制权。你只能等待AI完成工作，然后检查结果。如果结果不符合预期，你需要重新描述需求或手动修改代码。这种模式的问题在于，你无法在AI工作过程中进行干预，无法引导它朝着正确的方向前进。

Plan模式通过引入规划阶段，将AI的工作过程分为两步：先制定计划，再执行计划。在规划阶段，AI会详细列出它打算做什么、怎么做、按什么顺序做。你可以审查这个计划，提出修改意见，确认无误后再让AI执行。这种模式让你重新获得了对开发过程的控制权。

可控性的意义不仅在于防止AI走弯路，更在于它建立了一种人机协作的信任关系。当你能够看到AI的思考过程和执行计划时，你对AI的输出会更有信心。这种信任是AI编程工具被广泛采用的基础。


#### 9.1.2 透明性：让每一步都清晰可见

透明性是Plan模式的第二个核心理念。在直接执行模式下，AI的决策过程是不透明的。你不知道它为什么选择某种技术方案，为什么按照某种顺序执行，为什么修改了某些文件。这种不透明性不仅让人感到不安，也使得问题排查变得困难。

Plan模式通过详细的计划文档，让AI的决策过程完全透明。计划中会包含每一步操作的目的、涉及的文件、预期的变更内容，以及可能的风险点。你可以清楚地看到AI的思考逻辑，理解每一个决策背后的原因。

透明性还体现在执行过程中。Plan模式在执行每一步操作后，都会报告执行结果和下一步计划。你可以实时跟踪进度，了解当前处于哪个阶段，还剩多少工作未完成。这种实时的进度反馈，让你对整个开发过程了如指掌。


#### 9.1.3 纠错性：在错误发生之前就阻止它

纠错性可能是Plan模式最实用的价值。在直接执行模式下，如果AI理解错了你的需求，或者选择了不合适的技术方案，你往往需要等到它完成所有工作后才能发现问题。此时，纠正错误的成本可能很高，因为AI可能已经基于错误的方案生成了大量代码。

Plan模式通过规划阶段的审查机制，在执行之前就发现和纠正潜在的问题。当你审查AI生成的计划时，可以及时发现其中的偏差或风险，提出修改意见。这种先审查后执行的模式，将纠错的成本降到了最低。

从软件工程的角度来看，这实际上是一种左移策略。在软件开发中，越早发现和修复问题，修复成本越低。Plan模式将问题发现的时间点从执行阶段提前到了规划阶段，实现了质量保证的左移。

> “Plan模式的核心价值不在于让AI做得更快，而在于让AI做得更对。在软件工程中，做对事情比做快事情重要得多。” —— SOLO产品设计团队


### 9.2 Plan模式的工作流程

Plan模式的工作流程可以分为五个主要阶段：需求分析、计划生成、计划审查、计划执行和结果验证。每个阶段都有明确的目标和输出，形成了一个完整的闭环。


#### 9.2.1 阶段一：需求分析

当你选择Plan模式并输入需求后，SOLO首先会对你的需求进行深度分析。这个分析过程包括以下几个层面：

第一是需求理解。SOLO会解析你的需求描述，提取关键信息，包括功能需求、非功能需求、约束条件等。如果你的需求描述不够清晰，SOLO会主动提出澄清问题。

第二是项目扫描。SOLO会扫描当前项目的代码库，了解项目的结构、技术栈、编码风格等。这些信息对于制定合适的执行计划至关重要。

第三是依赖分析。SOLO会分析你的需求与现有代码之间的依赖关系，确定需要修改哪些文件、添加哪些新文件、是否需要引入新的依赖包等。

第四是风险评估。SOLO会评估执行这个需求可能面临的风险，包括技术风险（如兼容性问题）、范围风险（如需求蔓延）和回归风险（如影响已有功能）。


#### 9.2.2 阶段二：计划生成

在完成需求分析后，SOLO会生成一个详细的执行计划。这个计划通常包含以下内容：

| 计划要素 | 说明 | 示例 |
| --- | --- | --- |
| 目标概述 | 对要实现的功能的简明描述 | 为项目添加JWT用户认证功能 |
| 技术方案 | 采用的技术方案和理由 | 使用jsonwebtoken库实现JWT认证 |
| 执行步骤 | 详细的分步骤执行计划 | 1.安装依赖 2.创建认证中间件 3.添加登录接口... |
| 文件清单 | 需要创建或修改的文件列表 | 新建auth.js, 修改userRoutes.js, 修改server.js |
| 依赖变更 | 需要安装或更新的依赖包 | 安装jsonwebtoken, bcryptjs |
| 风险提示 | 可能的风险和应对措施 | 密码加密可能影响性能，建议使用异步加密 |
| 预估时间 | 预计执行时间 | 约3-5分钟 |


#### 9.2.3 阶段三：计划审查

计划生成后，SOLO会将完整的计划展示给你进行审查。你可以逐条查看计划的每个部分，对不满意的部分提出修改意见。审查过程中，你可以进行以下操作：

批准计划：如果你对计划完全满意，可以直接批准执行。SOLO会按照计划逐步执行所有操作。

修改计划：如果你对计划的某些部分有不同意见，可以提出修改。例如，你可能希望使用不同的技术方案，或者调整执行的顺序。SOLO会根据你的反馈修改计划。

补充需求：在审查计划的过程中，你可能会想到一些之前遗漏的需求。你可以随时补充，SOLO会将新需求整合到计划中。

拒绝计划：如果计划与你的预期差距太大，你可以拒绝并重新开始。SOLO会根据你的反馈重新分析需求并生成新的计划。


#### 9.2.4 阶段四：计划执行

计划批准后，SOLO开始按照计划逐步执行。执行过程中，每完成一步都会在界面上实时显示进度。你可以看到当前正在执行的操作、已完成的步骤和待执行的步骤。

如果在执行过程中遇到问题（如依赖安装失败、代码冲突等），SOLO会暂停执行并报告问题。你可以选择让SOLO自动修复、手动修复或中止执行。这种灵活的错误处理机制，确保了执行过程的可控性。


#### 9.2.5 阶段五：结果验证

所有步骤执行完成后，SOLO会进行结果验证。验证包括：检查所有计划的文件是否已创建或修改、运行自动化测试（如果项目中有）、检查代码是否有语法错误等。验证结果会以报告的形式展示给你，让你清楚地了解执行的结果。


### 9.3 实战案例：为项目添加用户登录功能

本节通过一个完整的实战案例，展示Plan模式的工作过程。我们将为一个已有的Node.js Express项目添加完整的用户登录功能，包括用户注册、登录认证、密码加密和JWT令牌管理。


#### 9.3.1 项目背景

假设我们有一个简单的待办事项（Todo）应用，使用Node.js + Express + MongoDB构建。目前应用只支持匿名使用，所有用户共享同一个待办事项列表。现在我们需要添加用户认证功能，让每个用户都有自己独立的待办事项。


#### 9.3.2 启动Plan模式

在SOLO中切换到Plan模式，输入以下需求：

💬 提示词：为现有的Todo应用添加完整的用户认证功能。具体要求：1) 用户注册接口（POST /api/auth/register），包含邮箱和密码，密码需要使用bcrypt加密存储；2) 用户登录接口（POST /api/auth/login），验证邮箱和密码，返回JWT token；3) 认证中间件，验证JWT token的有效性；4) 修改现有的Todo接口，要求用户登录后才能访问，每个用户只能看到和操作自己的待办事项；5) 添加用户模型（User Model），包含邮箱、密码哈希和创建时间；6) 所有接口需要基本的输入验证。请使用Mongoose作为MongoDB的ODM。


#### 9.3.3 SOLO生成的执行计划

SOLO经过分析后，生成了以下执行计划：

计划概览

目标：为Todo应用添加基于JWT的用户认证系统

技术方案：使用bcryptjs进行密码加密，jsonwebtoken进行JWT管理，express-validator进行输入验证

预估步骤：8步

预估时间：约4分钟

详细步骤

步骤1：安装依赖包

安装bcryptjs、jsonwebtoken、express-validator三个依赖包。这些包分别用于密码加密、JWT令牌管理和请求参数验证。

步骤2：创建User模型

在models目录下创建User.js文件，定义User Schema，包含email（唯一索引）、passwordHash和createdAt字段。添加密码哈希的pre-save钩子和密码验证的实例方法。

步骤3：创建认证工具模块

在utils目录下创建auth.js文件，封装JWT的生成和验证函数。包含generateToken和verifyToken两个函数，以及token过期时间配置。

步骤4：创建认证中间件

在middleware目录下创建authMiddleware.js文件，实现JWT验证中间件。从请求头中提取Bearer Token，验证其有效性，将用户信息附加到请求对象上。

步骤5：创建认证路由

在routes目录下创建authRoutes.js文件，实现注册和登录两个接口。注册接口包含邮箱格式验证和密码强度验证，登录接口返回JWT token。

步骤6：修改Todo路由

修改现有的todoRoutes.js，为所有Todo接口添加认证中间件。修改Todo的CRUD操作，确保每个用户只能访问自己的待办事项。

步骤7：更新服务器配置

修改server.js（或app.js），注册认证路由，添加JWT密钥的环境变量配置，添加全局错误处理中间件。

步骤8：创建.env配置文件

创建.env文件，添加JWT_SECRET、JWT_EXPIRES_IN、MONGODB_URI等环境变量配置。同时创建.env.example文件作为模板。

文件变更清单

| 操作类型 | 文件路径 | 说明 |
| --- | --- | --- |
| 新建 | models/User.js | 用户数据模型 |
| 新建 | utils/auth.js | JWT工具函数 |
| 新建 | middleware/authMiddleware.js | 认证中间件 |
| 新建 | routes/authRoutes.js | 认证路由 |
| 新建 | .env | 环境变量配置 |
| 新建 | .env.example | 环境变量模板 |
| 修改 | routes/todoRoutes.js | 添加认证和用户隔离 |
| 修改 | server.js | 注册新路由和中间件 |

风险提示

⚠️ 1. 修改Todo路由可能影响现有功能，建议先备份；2. JWT密钥需要使用强随机字符串，不能使用默认值；3. 密码哈希是CPU密集型操作，建议使用异步API；4. 需要确保MongoDB的唯一索引正确创建，防止重复注册。


#### 9.3.4 审查与确认

审查SOLO生成的计划后，我们可以发现计划非常全面，覆盖了从依赖安装到环境配置的所有环节。文件变更清单清晰列出了所有需要创建和修改的文件，风险提示也很有针对性。如果你对计划满意，可以点击批准并执行按钮。

假设你希望做一些调整，比如添加一个忘记密码的功能，你可以在审查阶段提出：

💬 提示词：计划整体很好，但请额外添加以下功能：1) 忘记密码接口（POST /api/auth/forgot-password），发送密码重置邮件；2) 重置密码接口（POST /api/auth/reset-password），验证重置令牌并设置新密码。请使用nodemailer发送邮件。

SOLO会根据你的反馈更新计划，添加两个新的步骤，并在文件变更清单中增加相应的文件。这种灵活的计划调整能力，是Plan模式的核心优势之一。

计划确认后，SOLO开始执行。你可以在界面上实时看到执行进度：

### 9.4 Plan模式的高级用法


#### 9.4.1 分阶段规划

对于大型需求，一次性规划所有步骤可能会导致计划过于复杂，难以审查。SOLO的Plan模式支持分阶段规划，将大型需求拆分为多个阶段，每个阶段独立规划和执行。

例如，为一个电商平台添加支付功能，可以拆分为以下阶段：

| 阶段 | 内容 | 依赖 |
| --- | --- | --- |
| 阶段一 | 设计支付系统架构和数据模型 | 无 |
| 阶段二 | 实现支付接口和安全中间件 | 阶段一 |
| 阶段三 | 集成第三方支付SDK | 阶段二 |
| 阶段四 | 添加支付回调和订单状态管理 | 阶段三 |
| 阶段五 | 编写测试用例和文档 | 阶段四 |

分阶段规划的好处是显而易见的。每个阶段的计划都比较简洁，容易审查和确认。同时，你可以在每个阶段完成后进行验证，确保前一个阶段的工作正确无误后再进入下一个阶段。这种渐进式的开发方式，大大降低了大型项目的风险。


#### 9.4.2 风险预判

Plan模式的一个高级特性是风险预判。SOLO在生成计划时，不仅会列出执行步骤，还会主动识别和评估潜在的风险。这些风险可能包括：

技术风险：如选择了不兼容的库版本、使用了过时的API、可能存在性能瓶颈等。SOLO会根据其训练数据中的项目经验，预判这些技术风险并给出建议。

架构风险：如修改可能破坏现有的架构设计、新增的模块与现有模块存在耦合等。SOLO会分析项目的架构依赖关系，识别可能的架构风险。

业务风险：如需求可能存在歧义、功能可能不符合用户预期等。SOLO会分析需求描述，识别模糊或矛盾之处，建议你进一步澄清。

💬 提示词：请为以下需求生成执行计划，并特别关注风险预判：将现有的REST API重构为GraphQL API。要求保持所有现有功能不变，同时添加GraphQL Playground用于调试。项目使用Express框架，MongoDB数据库。


### 9.5 Plan模式的最佳实践

要充分发挥Plan模式的价值，需要掌握一些最佳实践。这些实践来自大量用户的实际使用经验，能够帮助你更高效地使用Plan模式。


#### 9.5.1 需求描述的最佳实践

清晰的需求描述是Plan模式成功的基础。好的需求描述应该具备以下特征：

具体明确：避免模糊的描述，如优化性能，而应该具体说明将首页加载时间从3秒优化到1秒以内。

完整全面：列出所有相关的功能点和约束条件，避免遗漏。可以使用编号列表来组织需求，确保每个需求点都被覆盖。

上下文充分：提供足够的背景信息，让SOLO理解你的项目环境和约束条件。例如，说明项目使用的技术栈、团队编码规范、性能要求等。


#### 9.5.2 计划审查的最佳实践

审查计划时，应该重点关注以下几个方面：

技术方案是否合理：检查SOLO选择的技术方案是否符合你的预期和项目规范。如果你有特定的技术偏好，应该在需求描述中提前说明。

执行顺序是否正确：检查步骤的执行顺序是否合理。有些操作必须按特定顺序执行（如先创建模型再创建路由），而有些操作的顺序可以灵活调整。

风险是否被充分考虑：检查风险提示是否全面，是否有遗漏的风险点。如果你知道项目中存在一些特殊情况，应该在审查时提醒SOLO。


#### 9.5.3 迭代优化的最佳实践

Plan模式支持迭代优化。如果第一次生成的计划不够理想，不要急于拒绝，而是通过修改意见逐步完善。通常，经过一到两轮的迭代，就能得到一个高质量的计划。

💡 如果你发现SOLO反复生成不理想的计划，可能是需求描述本身存在问题。尝试重新组织需求描述，使用更结构化的格式，或者将大型需求拆分为多个小需求。


### 9.6 Plan模式 vs 直接执行：何时选择哪种

Plan模式和直接执行模式各有优劣，适用于不同的场景。本节将帮助你根据具体情况做出最佳选择。

| 考量维度 | Plan模式 | 直接执行模式 |
| --- | --- | --- |
| 需求复杂度 | 适合复杂、多步骤的需求 | 适合简单、单一步骤的需求 |
| 可控性要求 | 高——需要审查和确认每一步 | 低——信任AI的判断 |
| 时间紧迫度 | 适合不急于立即完成的任务 | 适合需要快速出结果的任务 |
| 风险容忍度 | 低——不能容忍错误 | 高——可以接受快速迭代 |
| 学习目的 | 适合学习AI的思考过程 | 适合快速获得结果 |
| 项目阶段 | 适合正式开发阶段 | 适合原型探索阶段 |
| 团队协作 | 适合需要多人审核的场景 | 适合个人快速开发 |
| 代码质量要求 | 高——需要确保每个决策正确 | 中——可以接受后续优化 |

一个实用的决策策略是：对于涉及多个文件修改、需要引入新依赖、或者可能影响现有功能的需求，优先使用Plan模式；对于简单的代码生成、单文件修改、或者探索性的开发任务，可以使用直接执行模式。

随着你对SOLO的信任度增加，你可能会越来越多地使用直接执行模式。但在面对重要的、不可逆的操作时，Plan模式始终是更安全的选择。

本章小结

Plan模式是SOLO正式版中体现可控AI理念的核心功能。通过将AI的工作过程分为规划和执行两个阶段，Plan模式在保持AI高效性的同时，赋予了用户对开发过程的完全控制权。可控性、透明性和纠错性三大设计理念，使得Plan模式特别适合复杂需求的开发。

通过为Todo应用添加用户认证功能的实战案例，我们完整体验了Plan模式从需求分析到结果验证的五个阶段。分阶段规划、风险预判等高级用法，进一步拓展了Plan模式的应用范围。掌握Plan模式的最佳实践，将帮助你更高效、更安全地利用AI进行软件开发。
