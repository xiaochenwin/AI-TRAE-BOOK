本章将通过一个完整的实战项目，带你体验使用 Trae 从零到一构建产品的全过程。我们将构建一个名为 'WriteFlow' 的 AI 写作助手，它能够帮助用户快速生成高质量的文章内容、智能改写已有文本、自动续写未完成的段落。这个项目将综合运用 Trae 的 MTC 模式、Code 模式、SOLO Builder 和 SOLO Coder 等所有核心功能。

通过这个项目，你将学到如何在真实的产品开发场景中高效运用 AI 编程工具，理解从产品构思到上线运营的完整链路，掌握每个环节中 Trae 的最佳使用方式。这不仅是一个技术实践，更是一次产品思维的全面锻炼。


### 16.1 项目背景与目标


#### 16.1.1 产品定位

WriteFlow 定位为一款面向内容创作者的 AI 写作助手工具。在当前内容为王的时代，无论是自媒体运营者、市场营销人员、学术研究者，还是日常需要大量文字工作的办公人员，都面临着写作效率低下的痛点。传统的写作方式需要耗费大量时间在构思、查阅资料、组织语言等环节上，而 AI 技术的发展为解决这一问题提供了全新的可能。

WriteFlow 的核心价值主张是：让 AI 成为你的写作搭档，而不是替代者。它不会替你完成所有写作工作，而是在你需要的环节提供智能辅助，帮助你突破写作瓶颈、提升内容质量、加快产出速度。这种定位既避免了用户对 AI 完全替代人类写作的担忧，又切实解决了效率痛点。

与市面上已有的 AI 写作工具相比，WriteFlow 的差异化优势在于：第一，支持多种写作场景，包括但不限于博客文章、营销文案、学术论文摘要、社交媒体内容等；第二，提供智能改写和续写功能，能够基于已有内容进行风格一致的扩展；第三，内置多种写作模板，降低使用门槛；第四，支持多语言内容生成与翻译。


#### 16.1.2 目标用户

WriteFlow 的目标用户可以分为以下几类：

核心用户群：自媒体创作者和内容运营人员。这类用户每天需要产出大量内容，对写作效率有极高的需求。他们通常需要撰写公众号文章、小红书笔记、抖音脚本等多种形式的内容，WriteFlow 可以帮助他们快速生成初稿、优化表达、调整风格。

扩展用户群：市场营销和品牌传播人员。这类用户需要频繁撰写广告文案、产品描述、新闻稿等商业内容。WriteFlow 的模板系统和风格定制功能可以确保输出内容符合品牌调性。

潜在用户群：学生和学术研究者。这类用户需要撰写论文、报告、摘要等学术内容。WriteFlow 可以帮助他们梳理思路、优化表达、检查语法。

长尾用户群：日常办公人员。这类用户偶尔需要撰写邮件、报告、总结等办公文档。WriteFlow 的低门槛设计可以让他们快速上手，提升日常写作效率。


#### 16.1.3 技术栈选型

在技术栈的选择上，我们需要综合考虑开发效率、可维护性、部署成本和扩展性等因素。借助 Trae 的 AI 辅助能力，我们可以更加自信地选择现代技术栈，因为 AI 能够帮助我们快速上手新技术并解决开发中的各种问题。

前端技术栈选择 Next.js 14 作为主框架，配合 React 18 和 TypeScript。Next.js 提供了优秀的开发体验、内置的 API 路由、服务端渲染能力和便捷的部署方案。UI 组件库选择 Tailwind CSS，它提供了原子化的样式方案，配合 shadcn/ui 组件库可以快速构建美观的界面。状态管理使用 Zustand，它比 Redux 更轻量，API 更简洁。

后端技术栈依托 Next.js 的 API Routes，配合 Prisma 作为 ORM 工具连接 PostgreSQL 数据库。认证系统使用 NextAuth.js，支持多种登录方式。AI 能力通过调用大语言模型 API 实现，包括 OpenAI GPT-4、Claude 等模型。

部署方案选择 Vercel 部署前端和 API，Railway 部署 PostgreSQL 数据库。这种组合既保证了性能和可靠性，又控制了成本，非常适合初创项目。

| 技术层级 | 技术选型 | 选择理由 |
| --- | --- | --- |
| 前端框架 | Next.js 14 + TypeScript | 全栈能力、SSR、开发体验好 |
| UI 样式 | Tailwind CSS + shadcn/ui | 原子化样式、组件丰富、可定制 |
| 状态管理 | Zustand | 轻量简洁、TypeScript 友好 |
| 数据库 | PostgreSQL + Prisma | 功能强大、类型安全、迁移方便 |
| 认证系统 | NextAuth.js | 开箱即用、多种登录方式 |
| AI 模型 | OpenAI GPT-4 / Claude | 文本生成能力强、API 稳定 |
| 前端部署 | Vercel | 零配置部署、全球 CDN、免费额度 |
| 数据库部署 | Railway | 简单易用、自动备份、按量计费 |

💡 在技术选型阶段，可以借助 Trae 的 MTC 模式进行技术方案的评估和对比。向 AI 描述你的项目需求，让它帮你分析不同技术栈的优劣，从而做出更明智的决策。


### 16.2 第一步：需求分析与PRD生成

需求分析是产品开发的第一步，也是最关键的一步。一个清晰、完整的需求文档能够为后续的开发工作奠定坚实的基础。在这一步中，我们将使用 Trae 的 MTC 模式来辅助完成需求分析和 PRD（产品需求文档）的生成。


#### 16.2.1 使用MTC模式进行需求梳理

打开 Trae，新建一个 MTC 模式的对话。在对话中，我们需要向 AI 清晰地描述产品愿景、目标用户和核心功能需求。以下是我们在 MTC 模式中使用的提示词：

💬 提示词：我正在开发一款名为 WriteFlow 的 AI 写作助手产品。请帮我梳理产品需求，生成一份完整的 PRD 文档。产品定位：面向内容创作者的 AI 写作辅助工具。核心功能包括：1) 文章生成 - 根据用户输入的主题和大纲生成完整文章；2) 智能改写 - 对已有文本进行风格转换、语言优化；3) 自动续写 - 基于已有内容智能续写后续段落；4) 写作模板 - 提供多种场景的写作模板。目标用户：自媒体创作者、市场营销人员、学生和办公人员。请从产品概述、用户故事、功能需求、非功能需求、数据需求五个维度来组织 PRD。

AI 会根据我们的描述生成一份结构化的 PRD 文档初稿。这份初稿通常包含了大部分我们需要的内容，但我们需要仔细审查并进行调整。重点关注以下几个方面：功能优先级是否合理、用户故事是否覆盖了核心场景、非功能需求是否全面、是否有遗漏的功能点。


#### 16.2.2 需求评审与迭代

生成初稿后，我们需要进行多轮的需求评审和迭代。在 MTC 模式中，我们可以针对具体的需求点进行深入讨论：

💬 提示词：关于文章生成功能，我需要更详细的需求描述。请帮我拆解这个功能，包括：输入参数（主题、大纲、风格、字数、语言等）、输出格式、生成流程、异常处理、性能要求。另外，请补充考虑多轮对话生成的场景，即用户可以在初次生成的基础上继续提出修改要求。

💬 提示词：请帮我补充 WriteFlow 的非功能需求，包括：1) 性能要求（响应时间、并发量）；2) 安全要求（数据加密、API 密钥管理）；3) 可用性要求（SLA、容灾方案）；4) 可扩展性要求（插件系统、模型切换）。请给出具体的量化指标。

经过多轮迭代，我们最终得到一份完整的 PRD 文档。这份文档将作为后续设计和开发工作的基准。在 Trae 中，我们可以将这份文档保存为项目文件，方便团队成员随时查阅。


#### 16.2.3 需求分析的关键产出

需求分析阶段的关键产出包括以下几个文档：

产品需求文档（PRD）：包含产品概述、目标用户画像、用户故事、功能需求列表（含优先级）、非功能需求、数据需求、版本规划等内容。这是整个项目最重要的参考文档。

功能清单：将所有功能点整理成清单形式，标注优先级（P0/P1/P2）和预计工作量。P0 功能是 MVP 必须包含的，P1 功能可以在第二个迭代中完成，P2 功能列入长期规划。

技术约束文档：记录技术选型的决策依据、已知的技术风险和应对方案、第三方服务的依赖关系等。

| 优先级 | 功能模块 | 功能点 | 预计工时 |
| --- | --- | --- | --- |
| P0 | 文章生成 | 基于主题和大纲生成文章 | 3天 |
| P0 | 智能改写 | 文本风格转换和语言优化 | 2天 |
| P0 | 用户系统 | 注册、登录、个人信息管理 | 2天 |
| P1 | 自动续写 | 基于上下文智能续写 | 2天 |
| P1 | 写作模板 | 多场景模板库 | 2天 |
| P1 | 收藏功能 | 文章收藏和历史记录 | 1天 |
| P2 | 多语言 | 内容翻译和多语言支持 | 3天 |
| P2 | 团队协作 | 多人协作编辑 | 5天 |

💡 在 MTC 模式中生成 PRD 时，建议分步骤进行。先生成大纲，再逐节填充详细内容。这样可以让 AI 在每个环节都保持专注，输出质量更高。


### 16.3 第二步：竞品调研与市场分析

在确定了产品需求之后，进行竞品调研和市场分析是必不可少的环节。这一步可以帮助我们了解市场格局、发现差异化机会、避免重复造轮子。同样，我们可以借助 Trae 的 MTC 模式来高效完成这项工作。


#### 16.3.1 竞品识别与分类

首先，我们需要识别出主要的竞品。AI 写作助手领域的竞品可以分为以下几类：

通用 AI 写作工具：如 Jasper、Copy.ai、Writesonic 等。这类工具功能全面，覆盖多种写作场景，但通常价格较高，且对中国市场的本地化支持不足。

垂直领域写作工具：如 Notion AI（知识管理场景）、Grammarly（语法检查场景）、秀米（公众号排版场景）等。这类工具在特定领域深耕，功能精准，但覆盖面较窄。

大模型直接应用：如 ChatGPT、Claude、文心一言等。用户直接与大模型对话来完成写作任务，灵活度高，但缺乏写作场景的专门优化。

国内 AI 写作产品：如秘塔写作猫、火山写作、阿里通义万相等。这类产品对中文写作场景有更好的支持，但在功能深度和用户体验上仍有提升空间。


#### 16.3.2 竞品分析框架

在 MTC 模式中，我们可以使用结构化的提示词来进行竞品分析：

💬 提示词：请帮我分析 AI 写作助手领域的竞品格局。重点关注以下维度：1) 核心功能对比（文章生成、改写、续写、模板等）；2) 定价策略对比（免费额度、付费方案、企业方案）；3) 目标用户群体差异；4) 技术实现方式（使用的 AI 模型、自研 vs 接入）；5) 用户体验亮点和不足。请以表格形式呈现对比结果，并给出 WriteFlow 的差异化定位建议。

AI 会生成一份详细的竞品分析报告。我们需要在此基础上进行验证和补充。特别是对于竞品的功能细节和定价信息，建议亲自体验竞品产品或查阅最新的公开资料来确认。

| 竞品 | 核心优势 | 主要不足 | 定价 |
| --- | --- | --- | --- |
| Jasper | 功能全面、品牌语音定制 | 价格高、中文支持弱 | $49/月起 |
| Copy.ai | 上手简单、模板丰富 | 生成质量不稳定 | $49/月起 |
| Notion AI | 与知识库深度集成 | 功能相对简单 | $10/月附加 |
| 秘塔写作猫 | 中文优化好、免费额度多 | 高级功能有限 | 免费+会员制 |
| ChatGPT | 灵活度高、能力全面 | 缺乏写作专门优化 | $20/月 |
| WriteFlow | 多场景、中文优化、性价比高 | 品牌知名度低 | 待定 |


#### 16.3.3 市场机会分析

通过竞品分析，我们可以识别出以下市场机会：

中文写作场景的深度优化：大部分海外竞品对中文写作的支持不够深入，包括中文的修辞习惯、排版规范、语义理解等方面。WriteFlow 可以在这些方面建立差异化优势。

性价比定位：当前市场上的 AI 写作工具价格普遍偏高，对于个人用户和小团队来说门槛较高。WriteFlow 可以通过合理的定价策略和免费额度来吸引这部分用户。

场景化模板：针对中国市场的特定写作场景（如公众号文章、小红书笔记、电商详情页等）提供专门的模板和优化，这是海外竞品难以覆盖的领域。

工作流集成：将 AI 写作能力与用户现有的工作流无缝集成，例如支持从大纲到成稿的完整工作流、支持与主流内容平台的对接等。

⚠️ 竞品分析不是一次性的工作，而应该持续进行。市场格局和竞品策略会不断变化，建议每月进行一次竞品动态追踪，及时调整产品策略。


### 16.4 第三步：原型设计与技术架构

在完成需求分析和竞品调研之后，下一步是进行原型设计和技术架构设计。这一步中，我们将使用 Trae 的 Code 模式来辅助完成设计工作，包括生成原型代码、设计技术架构文档等。


#### 16.4.1 原型设计

原型设计的目的是在正式开发之前，快速验证产品概念的可行性和用户体验的合理性。对于 WriteFlow，我们需要设计以下核心页面的原型：

首页/着陆页：展示产品价值主张、核心功能介绍、用户评价、定价方案和注册入口。这是用户接触产品的第一印象，需要精心设计。

写作工作台：这是产品的核心页面，包含输入区域（主题、大纲、关键词等）、生成按钮、输出展示区域、编辑工具栏、侧边栏（历史记录、收藏等）。

改写/续写页面：用户输入已有文本，选择改写或续写模式，配置参数（风格、长度等），查看和编辑结果。

模板库页面：展示所有可用的写作模板，支持分类筛选和搜索，每个模板卡片展示名称、描述、适用场景和预览。

用户中心页面：包含个人信息管理、API 密钥配置、使用量统计、订阅管理等功能。

在 Trae 的 Code 模式中，我们可以快速生成这些页面的原型代码：

💬 提示词：请帮我创建 WriteFlow 写作工作台页面的原型代码。使用 Next.js 14 + Tailwind CSS + shadcn/ui。页面布局要求：左侧是输入面板（包含主题输入框、大纲文本框、风格选择下拉框、字数滑块、生成按钮），右侧是输出展示区域（包含生成的文章内容、编辑工具栏、保存和复制按钮）。底部有改写和续写的快捷操作栏。请确保布局响应式，在移动端也能正常使用。

AI 生成的原型代码可以快速在浏览器中预览，让我们直观地看到页面的效果。如果对某些细节不满意，可以直接在对话中提出修改要求，AI 会即时调整代码。这种快速迭代的方式比传统的原型设计工具效率高出数倍。


#### 16.4.2 技术架构设计

技术架构设计是确保项目可维护性和可扩展性的关键环节。在 Trae 的 Code 模式中，我们可以让 AI 帮助生成技术架构文档和项目结构：

💬 提示词：请帮我设计 WriteFlow 的技术架构。要求：1) 前端使用 Next.js 14 App Router 架构，包含页面路由和 API 路由；2) 数据库使用 PostgreSQL + Prisma ORM，设计核心数据模型（User、Article、Template、Favorite 等）；3) AI 服务层抽象设计，支持多模型切换（OpenAI、Claude、国产模型）；4) 认证系统使用 NextAuth.js，支持邮箱密码和 GitHub OAuth 登录；5) 请给出完整的项目目录结构、数据库 Schema 和 API 接口设计。

AI 会生成一份详细的技术架构文档，包含项目目录结构、数据库模型定义、API 接口列表等内容。以下是一个精简版的数据库 Schema 设计：

// prisma/schema.prisma

model User {

id String @id @default(cuid())

email String @unique

name String?

avatar String?

articles Article[]

favorites Favorite[]

createdAt DateTime @default(now())

updatedAt DateTime @updatedAt

}

model Article {

id String @id @default(cuid())

title String

content String @db.Text

type ArticleType // GENERATE, REWRITE, CONTINUE

userId String

user User @relation(fields: [userId])

templateId String?

template Template? @relation(fields: [templateId])

createdAt DateTime @default(now())

}

model Template {

id String @id @default(cuid())

name String

description String @db.Text

category String

prompt String @db.Text

articles Article[]

isPublic Boolean @default(true)

}

在技术架构中，API 接口设计是连接前后端的桥梁。以下是 WriteFlow 的核心 API 接口设计：

| 接口 | 方法 | 路径 | 说明 |

💡 在 Code 模式中设计技术架构时，建议让 AI 生成多个备选方案进行对比。例如，可以分别让 AI 设计单体架构和微服务架构的方案，然后根据项目规模和团队能力选择最合适的方案。


### 16.5 第四步：项目初始化与核心开发

#### 16.6.1 优化生成流程

SOLO Builder 生成的基础文章生成功能虽然可以工作，但在用户体验上还有很大的优化空间。我们需要实现以下改进：

流式输出：文章生成通常需要几秒到十几秒的时间，如果等生成完成后一次性展示，用户等待体验很差。我们需要实现流式输出，让文字像打字一样逐步显示。

生成参数配置：除了主题和大纲，用户还应该能够配置更多参数，如写作风格（正式/轻松/幽默）、目标读者（专业/大众/青少年）、文章长度（短/中/长）、是否包含标题和摘要等。

生成历史：每次生成都应该保存为一条记录，方便用户回顾和比较不同版本的生成结果。

在 SOLO Coder 中，我们逐步实现这些优化：

💬 提示词：请帮我优化 WriteFlow 的文章生成功能，实现流式输出。要求：1) API 路由 /api/articles/generate 使用 Server-Sent Events (SSE) 返回流式数据；2) 前端使用 EventSource 或 fetch + ReadableStream 接收流式数据；3) 在生成过程中显示加载动画和已生成的文字；4) 生成完成后自动保存到数据库；5) 支持用户在生成过程中点击停止按钮中断生成。

流式输出的实现涉及前后端的配合。后端需要将 OpenAI 的流式响应转发给前端，前端需要逐步更新 UI。以下是后端的核心实现逻辑：

// app/api/articles/generate/route.ts

export async function POST(req: Request) {

const { topic, outline, style, length } = await req.json();

const encoder = new TextEncoder();

const stream = new ReadableStream({

async start(controller) {

try {

const aiService = createAIService();

await aiService.generateStream(

{ topic, outline, style, length },

(chunk) => {

controller.enqueue(

encoder.encode(`data: ${JSON.stringify(chunk)}\n\n`)

);

}

);

controller.enqueue(encoder.encode('data: [DONE]\n\n'));

} catch (error) {

controller.enqueue(

encoder.encode(`data: ${JSON.stringify({error: error.message})}\n\n`)

);

} finally {

controller.close();

}

}

});

return new Response(stream, {

headers: { 'Content-Type': 'text/event-stream' }

});

}

### 16.7 第六步：功能迭代——智能改写与续写

智能改写和续写是 WriteFlow 的另外两个核心功能。改写功能帮助用户优化已有文本的表达方式，续写功能则基于已有内容智能生成后续段落。这两个功能在实现上有相似之处，都需要理解上下文语义并生成连贯的内容。


#### 16.7.1 智能改写功能

智能改写功能支持多种改写模式，满足用户在不同场景下的需求：

风格转换：将文本从一种风格转换为另一种风格，例如从正式风格转换为轻松风格，从中文转换为英文等。

语言优化：在不改变原文含义的前提下，优化语言表达，使其更加流畅、精炼或专业。

扩写缩写：对文本进行扩写（增加细节和论述）或缩写（提炼核心要点）。

纠错润色：修正文本中的语法错误、错别字、标点符号等问题，同时优化表达。

在 SOLO Coder 中实现改写功能：

💬 提示词：请帮我实现 WriteFlow 的智能改写功能。要求：1) 支持四种改写模式：风格转换、语言优化、扩写缩写、纠错润色；2) 每种模式有不同的配置选项（如风格转换可选目标风格，扩写缩写可选目标长度比例）；3) 改写结果与原文并排展示，方便用户对比；4) 支持接受改写结果、重新改写、手动编辑三种操作；5) 改写历史记录保存在数据库中。

#### 16.7.2 自动续写功能

自动续写功能需要深入理解已有内容的上下文，生成语义连贯、风格一致的后续内容。这比从零开始生成文章更具挑战性，因为 AI 需要准确把握已有内容的主题、语气、结构和未表达完的意思。

💬 提示词：请帮我实现 WriteFlow 的自动续写功能。要求：1) 用户输入已有文本，系统分析上下文并生成续写内容；2) 支持配置续写长度（短：100-200字、中：200-500字、长：500-1000字）；3) 续写内容与原文风格和语气保持一致；4) 支持多轮续写，每次续写基于之前所有内容；5) 续写结果可以一键合并到原文中，也可以单独保存。

续写功能的实现要点包括：上下文窗口管理（确保发送给 AI 的上下文不超过模型限制）、风格一致性保持（通过系统提示词和 few-shot 示例）、续写长度的精确控制（通过 token 计数和停止条件）。


#### 16.7.3 改写与续写的交互设计

改写和续写功能在交互设计上有一些共同的特点和挑战：

对比展示：改写功能需要将原文和改写结果并排展示，续写功能需要清晰标识哪些是原文、哪些是续写内容。这种对比展示需要精心设计布局，确保在不同屏幕尺寸下都有良好的可读性。

版本管理：用户可能对同一段文本进行多次改写或续写，需要提供版本管理功能，让用户可以查看和比较不同版本。

撤销与重做：提供撤销和重做操作，让用户可以自由探索不同的改写和续写方案，而不必担心丢失之前的结果。

⚠️ 在实现改写和续写功能时，需要特别注意内容安全。建议添加内容审核机制，过滤可能包含有害、违规或不当内容的生成结果。可以使用 AI 模型自带的审核功能，或者接入第三方内容审核 API。

#### 16.8.2 收藏功能实现

收藏功能允许用户标记自己喜欢的文章或模板，方便后续快速访问。实现收藏功能需要考虑数据模型设计和交互体验两个方面。

数据模型方面，收藏关系是一个典型的多对多关系。用户可以收藏多篇文章，一篇文章也可以被多个用户收藏。在 Prisma 中，这种关系通过中间表来实现：

model Favorite {

id String @id @default(cuid())

userId String

user User @relation(fields: [userId])

articleId String

article Article @relation(fields: [articleId])

createdAt DateTime @default(now())

@@unique([userId, articleId])

}

交互体验方面，收藏功能需要做到即时反馈。当用户点击收藏按钮时，应该立即更新 UI 状态（按钮变为已收藏状态），同时异步发送请求到后端。如果网络请求失败，需要回滚 UI 状态并提示用户。这种乐观更新的策略可以显著提升用户体验。

💬 提示词：请帮我实现 WriteFlow 的收藏功能。要求：1) 在文章列表和文章详情页添加收藏按钮（心形图标），支持切换收藏状态；2) 使用乐观更新策略，点击后立即更新 UI，异步同步到后端；3) 用户中心添加收藏列表页面，支持分页加载和取消收藏；4) 收藏数据使用 localStorage 缓存，减少 API 请求；5) 收藏按钮有动画效果（缩放+颜色变化）。

前端性能：使用 Next.js 的内置优化功能，包括图片优化（next/image）、字体优化（next/font）、代码分割和懒加载。使用 Lighthouse 进行性能审计，确保 LCP（最大内容绘制）小于 2.5 秒，FID（首次输入延迟）小于 100 毫秒。

API 性能：对频繁访问的 API 添加缓存策略，使用 Redis 或 Next.js 的内置缓存。对数据库查询进行优化，添加适当的索引。对 AI API 调用实现请求队列和速率限制。

AI 生成性能：使用流式输出来减少用户感知的等待时间。实现请求缓存，对相同的输入返回缓存的结果。优化提示词长度，减少不必要的 token 消耗。

💬 提示词：请帮我优化 WriteFlow 的前端性能。要求：1) 使用 next/dynamic 对写作工作台组件进行懒加载；2) 使用 React.memo 和 useMemo 优化组件渲染性能；3) 实现虚拟滚动来优化文章列表的渲染性能（大量文章时）；4) 添加 Service Worker 实现离线缓存；5) 优化图片和字体的加载策略。


#### 16.9.3 安全性检查

安全性是 Web 应用不可忽视的方面。以下是 WriteFlow 需要关注的安全措施：

输入验证和消毒：对所有用户输入进行严格的验证和消毒，防止 XSS 和 SQL 注入攻击。使用 Zod 进行输入验证，使用 DOMPurify 对 HTML 内容进行消毒。

API 密钥安全：AI 模型的 API 密钥只能在服务端使用，绝不能暴露到前端。使用环境变量管理密钥，并定期轮换。

速率限制：对所有 API 接口实施速率限制，防止滥用和 DDoS 攻击。可以使用 upstash/ratelimit 或类似的库来实现。

CORS 配置：严格配置 CORS 策略，只允许来自可信域名的请求。

内容安全策略（CSP）：配置 CSP 头部，限制页面可以加载的资源来源，防止 XSS 攻击。

⚠️ 安全性检查不是一次性的工作，而应该贯穿整个开发周期。建议在 CI/CD 流程中集成安全扫描工具（如 Snyk、OWASP ZAP），在每次代码提交时自动进行安全检查。


### 16.10 第九步：部署上线

经过充分的开发和测试，WriteFlow 已经准备好部署上线了。我们将使用 Vercel 部署前端应用和 API，使用 Railway 部署 PostgreSQL 数据库。这种组合提供了简单、可靠且具有良好扩展性的部署方案。


#### 16.10.1 数据库部署（Railway）

首先，我们需要在 Railway 上部署 PostgreSQL 数据库：

第一步：访问 Railway 官网（railway.app），使用 GitHub 账号登录。

第二步：点击 'New Project'，选择 'Provision PostgreSQL'。Railway 会自动创建一个 PostgreSQL 数据库实例。

第三步：在数据库服务的 'Variables' 标签页中，找到 DATABASE_URL 连接字符串，复制保存。

第四步：（可选）在 'Settings' 标签页中配置自动备份策略，建议每天备份一次。

数据库部署完成后，我们需要运行数据库迁移来创建表结构。可以通过 Railway 的 CLI 工具来执行：

# 安装 Railway CLI

npm install -g @railway/cli

# 登录 Railway

railway login

# 连接到项目

railway link

# 运行数据库迁移

railway run npx prisma migrate deploy


#### 16.10.2 前端部署（Vercel）

Vercel 是 Next.js 的官方部署平台，提供了零配置的部署体验。部署步骤如下：

第一步：将项目代码推送到 GitHub 仓库。确保 .gitignore 文件中包含了 .env.local 和 node_modules。

第二步：访问 Vercel 官网（vercel.com），使用 GitHub 账号登录。

第三步：点击 'New Project'，选择刚才推送的 GitHub 仓库。

第四步：在 'Environment Variables' 中配置以下变量：DATABASE_URL（Railway 提供的连接字符串）、NEXTAUTH_SECRET（生成一个随机字符串）、NEXTAUTH_URL（你的域名）、OPENAI_API_KEY、GITHUB_ID、GITHUB_SECRET。

第五步：点击 'Deploy'，等待构建和部署完成。Vercel 会自动检测 Next.js 项目并配置构建命令。

部署完成后，Vercel 会提供一个默认域名（如 writeflow.vercel.app）。你可以配置自定义域名：在项目设置中添加你的域名，然后按照提示在域名服务商处配置 DNS 记录。

健康检查：设置 /api/health 端点，配合 UptimeRobot 等服务进行可用性监控。

💬 提示词：请帮我为 WriteFlow 创建一个健康检查 API 端点 /api/health。要求：1) 检查数据库连接是否正常；2) 检查 AI 服务 API 密钥是否配置；3) 返回系统状态（OK/DEGRADED/DOWN）、各组件状态、响应时间等信息；4) 使用适当的 HTTP 状态码（200/503）。

#### 16.12.1 Trae 使用经验总结

MTC 模式的最佳使用场景：适合在项目的早期阶段进行需求分析、方案设计和文档编写。MTC 模式的对话式交互非常适合探索性的工作，可以帮助我们快速理清思路、生成结构化的文档。但需要注意的是，MTC 模式生成的文档需要人工审查和调整，不能完全依赖 AI 的输出。

Code 模式的最佳使用场景：适合在技术方案确定后进行代码实现和问题排查。Code 模式可以直接操作项目文件，实现精准的代码修改。在使用 Code 模式时，提供足够的上下文信息（相关文件、错误日志、期望行为）是获得高质量输出的关键。

SOLO Builder 的最佳使用场景：适合在项目初始化阶段快速搭建项目框架。SOLO Builder 可以在几分钟内生成一个功能完整的项目脚手架，大大缩短了项目启动时间。但 SOLO Builder 生成的代码需要仔细审查，特别是安全性、错误处理和边界情况等方面。

SOLO Coder 的最佳使用场景：适合在项目迭代阶段进行功能开发和优化。SOLO Coder 可以理解项目的整体架构，在添加新功能时保持代码风格和架构的一致性。建议将大的功能需求拆分成小的任务，逐个使用 SOLO Coder 实现。


#### 16.12.2 项目管理经验

需求管理：在项目初期花足够的时间进行需求分析和梳理，可以避免后期的返工。使用 PRD 文档作为需求的唯一来源，确保所有开发工作都有明确的需求依据。

技术债务管理：在快速开发的过程中不可避免地会积累技术债务。建议每个迭代预留 20% 的时间来处理技术债务，包括代码重构、测试补充、文档更新等。

质量保障：自动化测试是质量保障的基石。从项目初期就建立测试体系，随着项目的发展不断补充测试用例。CI/CD 流程中的自动化测试可以在代码合并前发现问题，防止缺陷流入生产环境。
