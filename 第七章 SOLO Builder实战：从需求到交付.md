### 7.1 SOLO Builder的工作原理


#### 7.1.1 五阶段工作流

SOLO Builder的工作流程分为五个阶段，每个阶段都有明确的输入、处理和输出。理解这五个阶段，有助于你更好地与Builder协作。

阶段一：需求解析与扩展。Builder接收你的需求描述后，首先进行深度解析。它不仅理解字面意思，还会推断隐含需求。例如，你要求"创建一个博客系统"，Builder会自动推断你需要文章管理、用户系统、评论功能、搜索功能等。解析完成后，Builder会向你展示它的理解，确认无误后进入下一阶段。

阶段二：方案设计与规划。确认需求后，Builder会制定详细的技术方案。这包括技术栈选择、目录结构设计、数据模型设计、API设计、UI设计等。Builder会参考业界最佳实践和你的项目特点，制定最适合的方案。方案设计完成后，Builder会展示方案概要，征求你的意见。

阶段三：代码实现。方案确认后，Builder开始逐步实现代码。它会按照依赖关系的顺序创建文件——先创建配置和基础模块，再创建核心业务逻辑，最后创建UI组件。每个文件的创建都会在AI面板中实时显示，你可以随时查看和干预。

阶段四：验证与调试。代码实现完成后，Builder会自动进行验证。它运行构建命令检查语法错误，安装依赖确保环境正确，启动开发服务器验证功能。如果发现问题，Builder会自主修复并重新验证。

阶段五：优化与交付。验证通过后，Builder会对代码进行优化——改善性能、增强错误处理、添加注释和文档。最后，Builder会提供一个完整的项目摘要，包括项目结构、功能清单、使用说明和后续建议。


#### 7.1.2 Builder的决策机制

Builder在执行过程中需要做出大量的决策：使用哪个框架？如何组织代码？如何处理边界情况？这些决策基于三个信息源。

第一，需求描述中的明确要求。如果你在需求中指定了技术栈，Builder会优先遵循。第二，项目上下文中的隐含信息。CUE引擎会分析项目中已有的代码，推断技术偏好和编码规范。第三，Builder的训练数据和最佳实践知识。当需求描述和项目上下文都没有明确指示时，Builder会参考其训练数据中学到的最佳实践。

💡 如果你对Builder的某个决策不满意，可以在任何时候通过对话来纠正。Builder会根据你的反馈调整后续的决策和实现。


### 7.2 实战案例一：图片格式转换网站


#### 7.2.1 场景描述

图片格式转换是一个常见的开发需求。本案例将展示如何用SOLO Builder构建一个功能完整的在线图片格式转换网站。这个网站支持多种图片格式之间的相互转换（PNG、JPG、WebP、BMP、GIF），提供批量转换功能，并具有友好的用户界面。


#### 7.2.2 步骤一：描述需求

💬 提示词：创建一个在线图片格式转换网站。功能要求：1) 支持PNG、JPG、WebP、BMP、GIF格式互转；2) 支持拖拽上传和点击上传；3) 支持批量转换（最多20张）；4) 显示转换进度和结果预览；5) 一键下载转换后的图片；6) 支持调整输出质量（针对JPG和WebP）；7) 响应式设计，支持移动端使用。技术要求：前端使用React + Tailwind CSS，后端使用Node.js + Express，图片处理使用Sharp库。


#### 7.2.3 步骤二：Builder的规划

Builder接收到需求后，首先进行了需求解析。它将需求分解为以下功能模块：

转换模块：使用Sharp库进行图片格式转换，支持质量参数调整。

进度管理模块：跟踪批量转换的进度，提供实时进度反馈。

下载模块：提供单张下载和批量打包下载功能。

UI模块：响应式界面设计，包括上传区域、转换选项、进度显示、结果预览。


#### 7.2.4 步骤三：核心代码实现

Builder生成的后端核心转换逻辑如下：

const sharp = require("sharp");

const path = require("path");

async function convertImage(inputPath, outputFormat, quality) {

const outputDir = path.join(__dirname, "../uploads/converted");

const outputName = `${Date.now()}-${Math.random().toString(36).slice(2)}.${outputFormat}`;

const outputPath = path.join(outputDir, outputName);

let pipeline = sharp(inputPath);

if (outputFormat === "jpeg" || outputFormat === "jpg") {

pipeline = pipeline.jpeg({ quality: quality || 85 });

} else if (outputFormat === "png") {

pipeline = pipeline.png({ quality: quality || 100 });

} else if (outputFormat === "webp") {

pipeline = pipeline.webp({ quality: quality || 85 });

} else if (outputFormat === "bmp") {

pipeline = pipeline.bmp();

}

await pipeline.toFile(outputPath);

return outputName;

}

Builder生成的前端上传组件使用了拖拽API和文件读取API，实现了流畅的拖拽上传体验。质量调整功能使用了滑块控件，实时显示预览效果。


#### 7.2.5 步骤四：测试与部署

Builder在完成代码实现后，自动进行了以下验证：安装了所有依赖包（sharp、express、multer、archiver等），运行了构建检查，启动了开发服务器，并在浏览器中测试了所有功能。

测试过程中，Builder发现了一个边界情况：当用户上传非图片文件时，Sharp库会抛出异常。Builder自动添加了文件类型验证逻辑，在服务端对上传文件的MIME类型进行检查，确保只有图片文件才能被处理。

⚠️ Sharp库在安装时需要编译原生模块，在某些环境下可能会遇到安装失败的问题。如果遇到这种情况，可以尝试使用sharp的预编译版本，或者参考Sharp官方文档的故障排除指南。

### 7.5 SOLO Builder的最佳实践


#### 7.5.1 提示词技巧

好的提示词是获得高质量产出的关键。以下是一些经过验证的提示词技巧。

技巧一：明确功能需求。列出你想要的所有功能，不要假设AI能猜到你的需求。功能列表越详细，Builder的产出越符合你的期望。

技巧二：指定技术栈。如果你对技术栈有偏好，在提示词中明确指定。例如"使用Vue 3 + TypeScript + Pinia"。如果不指定，Builder会根据项目特点自动选择，但可能不符合你的偏好。

技巧三：描述UI风格。如果你对界面设计有特定要求，在提示词中描述。例如"使用深色主题"、"界面风格简约现代"、"参考Apple的设计语言"。

技巧四：说明约束条件。列出项目的约束条件，如"不支持IE浏览器"、"需要支持国际化"、"数据量可能达到百万级"。这些约束会影响Builder的技术决策。

技巧五：分步骤构建。对于复杂项目，不要试图一次性描述所有需求。先描述核心功能，让Builder生成基础版本，然后逐步添加新功能。这种迭代式的方式通常比一次性构建更高效。


#### 7.5.2 交互技巧

与SOLO Builder的交互也需要一些技巧。

及时确认和纠正。当Builder向你展示它的理解或方案时，及时确认或纠正。早期的纠正比后期的修改成本低得多。

善用@引用。在对话中使用@引用来指向特定的文件或代码片段，确保Builder准确理解你讨论的对象。

提供示例代码。如果你有特定的代码风格或实现方式偏好，可以提供示例代码让Builder参考。

分阶段验收。不要等到Builder完成所有工作后才验收。在每个阶段完成后进行检查，发现问题及时反馈。


#### 7.5.3 注意事项

⚠️ 注意事项一：代码安全。Builder生成的代码可能包含安全漏洞，特别是涉及用户输入、文件操作和网络请求的部分。在部署到生产环境前，务必进行安全审查。

⚠️ 注意事项二：性能考量。Builder生成的代码通常以功能正确为首要目标，可能不会进行充分的性能优化。对于性能敏感的应用，建议在Builder生成代码后进行性能分析和优化。

⚠️ 注意事项三：依赖管理。Builder会自动安装项目依赖，但可能不会选择最优的版本。建议检查package.json中的依赖版本，确保没有已知的安全漏洞或兼容性问题。

⚠️ 注意事项四：数据隐私。如果你的应用涉及敏感数据，确保Builder生成的数据处理逻辑符合隐私保护要求。特别注意localStorage的使用——敏感数据不应该存储在客户端。


### 7.6 常见问题与解决方案

以下是开发者在使用SOLO Builder时最常遇到的问题及其解决方案。

| 问题 | 原因分析 | 解决方案 |
| --- | --- | --- |
| Builder生成的代码风格不一致 | CUE未充分学习项目模式 | 在项目中添加代码风格示例文件，或使用.eslintrc等配置文件 |
| Builder在大型项目中表现下降 | 上下文窗口限制 | 将大项目拆分为多个子项目，或使用@引用指定相关文件 |
| Builder忽略了部分需求 | 需求描述过长或不够明确 | 将需求分步骤描述，每步聚焦一个功能模块 |
| 构建失败但代码看起来正确 | 依赖版本不兼容 | 检查package.json中的依赖版本，尝试使用兼容版本 |
| UI效果与预期差距大 | 提示词中的UI描述不够具体 | 提供UI参考图或更详细的样式描述 |
| Builder修改了不该修改的文件 | CUE的依赖分析有误 | 在对话中明确告知Builder不要修改特定文件 |
| 生成的代码缺少错误处理 | Builder倾向于生成最简实现 | 在需求中明确要求"完善的错误处理" |
| 移动端适配不佳 | Builder默认优先桌面端 | 在需求中明确要求"移动端优先"或"响应式设计" |

除了表格中的问题，开发者还应该注意以下几点。首先，SOLO Builder虽然强大，但它不是万能的。对于需要深度领域知识的特定业务逻辑，人类的判断和经验仍然是不可替代的。其次，Builder生成的代码应该被视为"第一稿"而非"最终版"。它为你提供了一个高质量的起点，但后续的优化和打磨仍然需要人类的参与。
